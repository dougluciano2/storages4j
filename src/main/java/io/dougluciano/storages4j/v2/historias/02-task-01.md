# Task 02 — Criar o contrato do provider client

## 🎯 Objetivo
Definir a interface responsável pela comunicação de baixo nível com o provider de storage, separando a infraestrutura da regra de negócio da biblioteca.

Essa interface deve representar a porta de saída da arquitetura para operações técnicas no storage.

---

## 📦 Pacote
io.dougluciano.storages4j.v2.provider

---

## 📛 Nome sugerido
StorageProviderClient

---

## 🧩 Responsabilidade
A interface deve conter apenas operações técnicas e agnósticas de vendor, sem qualquer regra de negócio.

Ela deve ser responsável por:

- enviar objeto ao storage
- recuperar objeto do storage
- remover objeto
- verificar existência de objeto
- consultar metadados técnicos do objeto
- verificar existência do bucket
- criar bucket, se necessário

---

## 🛠️ Métodos sugeridos

- putObject(String objectKey, InputStream inputStream, long sizeBytes, String contentType)
- getObject(String objectKey)
- removeObject(String objectKey)
- exists(String objectKey)
- stat(String objectKey)
- bucketExists()
- createBucketIfNotExists()

---

## ⚠️ Regras importantes

- Não deve conter lógica de metadata
- Não deve conter lógica de UUID
- Não deve conter decisões baseadas em configuração
- Não deve conhecer fluxo de upload completo
- Não deve expor classes do SDK do provider no contrato
- Deve trabalhar apenas com tipos da própria biblioteca e tipos nativos do Java

---

## 🧠 Diretriz arquitetural

Essa interface representa o adapter/port de infraestrutura da biblioteca.

Ela será implementada por classes concretas, como por exemplo:

- MinioStorageProviderClient

No futuro, outras implementações poderão existir sem impactar a camada de serviço.

---

## 🎯 Resultado esperado

- Interface criada e bem definida
- Contrato neutro e reutilizável
- Base pronta para implementação concreta do provider MinIO
- Separação clara entre infraestrutura e orquestração
